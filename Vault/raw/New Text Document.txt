Good - this is the point where serious robotics projects either stay messy or become engineering systems.

For the Tier 3 robot system, the planning scope is:

- Camera
- ORB-SLAM3
- IMU
- Wheel encoders
- A* path planning
- PID motor control
- YOLO object detection later

The vault should contain a small set of engineering-style planning documents.

## 1. System Architecture Document

Purpose: define how everything connects.

Required content:

- Hardware blocks
- Software blocks
- Data flow
- Decision flow

Core flow:

```text
Sensors -> Processing -> Mapping -> Planning -> Control -> Motors
```

Required connections:

- Camera -> SLAM input
- IMU -> orientation correction
- Encoders -> odometry
- SLAM -> pose + map
- A* -> path planning
- PID -> motor control

## 2. Hardware Design Document

Purpose: define the physical robot setup.

Required content:

- Motor type: DC motors with encoders
- Wheel diameter
- Wheelbase
- IMU placement
- Camera angle and height
- Battery system
- Wiring diagram

Example values:

```text
Wheel diameter: 6.5 cm
Wheelbase: 14 cm
Camera tilt: 15 degrees
IMU mounted center of chassis
```

## 3. Sensor & Calibration Document

Purpose: explain how sensor measurements become reliable.

Required content:

- Encoder ticks per revolution
- Centimeters per tick formula
- IMU drift calibration
- Gyro bias correction
- Camera resolution
- Camera FPS
- Lens distortion handling

## 4. SLAM Configuration Document

Purpose: define the ORB-SLAM3 setup.

Required content:

- Feature settings
- Camera intrinsics
- IMU noise parameters
- Frame-rate synchronization

Example values:

```text
Camera FPS: 30
IMU rate: 200 Hz
Feature points: 1000-2000
```

## 5. Navigation & Planning Document

Purpose: define how the robot moves from A to B.

Required content:

- Occupancy grid format
- A* algorithm rules
- Cost functions
- Goal selection

Example costs:

```text
Obstacle cost = 100
Free space = 1
Unknown space = 50
```

## 6. Control System Document

Purpose: define how the robot physically moves.

Required content:

- PID controller design
- Motor mixing
- Speed limits
- Turning behavior

Example values:

```text
Kp = 1.2
Ki = 0.01
Kd = 0.3
```

## 7. Software Architecture Document

Purpose: define the code structure.

Required modules:

```text
/slam
/planning
/control
/sensors
/motors
/camera
```

Required content:

- Thread ownership
- 30 FPS camera loop
- 200 Hz IMU loop
- Communication between modules

## 8. Testing & Validation Plan

Purpose: define how the system proves it works.

Required tests:

- Straight-line encoder test
- 90-degree IMU turn test
- SLAM drift test
- Mapping accuracy test
- Navigation A-to-B test

Example requirement:

```text
Robot must travel 2 meters within +/- 2 cm error.
```

## 9. Failure Handling Document

Purpose: define what happens when subsystems fail.

Required cases:

- Lost SLAM tracking
- Encoder mismatch
- IMU drift
- Sudden obstacle appearance

Example rule:

```text
If SLAM tracking is lost, switch to IMU + encoder dead reckoning.
```

## Final structure

Minimum set:

1. System Architecture
2. Hardware Design
3. Navigation with A*
4. Control with PID

Advanced set:

5. SLAM Configuration
6. Sensor Calibration
7. Software Architecture
8. Testing Plan
9. Failure Handling
